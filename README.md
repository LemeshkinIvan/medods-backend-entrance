# Task Service

Сервис для управления задачами с поддержкой периодических правил выполнения.  
Реализован REST API на Go.

---

## 📌 Особенности

Сервис поддерживает задачи с различными типами периодичности:

- ежедневные (каждый N-й день)
- ежемесячные (на конкретное число месяца)
- чётные / нечётные дни месяца
- конкретные даты (custom dates)

Задачи не дублируются в базе данных — применяется rule-based подход:  
активность задачи вычисляется динамически при получении списка.

## Работа с периодичностью
Поддерживается фильтрация задач по дате:
GET /api/v1/tasks?date=DD.MM.YYYY
Если date не передан — возвращаются все задачи без фильтрации по дате.

## Основные маршруты
```
POST /api/v1/tasks
GET /api/v1/tasks
GET /api/v1/tasks/{id}
PUT /api/v1/tasks/{id}
DELETE /api/v1/tasks/{id}
```
## Подход к реализации

При запросе списка задач:

Валидация входных данных при создании/обновлении задачи:
 - проверка корректности createdAt / scheduledAt
 - проверка согласованности custom_dates для RepetitionSpecific
 - Получение задач из базы данных
 - Фильтрация по диапазону дат (created_at / scheduled_at) на уровне БД
 - Применение правил периодичности:
    - daily (каждый N день)
    - monthly (по числу месяца)
    - even/odd days
    - custom_dates

Активность задачи вычисляется динамически при запросе списка,
в зависимости от переданной даты и правил периодичности.

### 🔄 Альтернативные подходы

1. Pre-generation (создание задач заранее)
Подход, при котором система заранее генерирует все экземпляры задач на заданный период (например, месяц или год вперёд).

❌ Недостатки:
 - рост объёма данных при большом количестве задач
 - проблема “бесконечных” задач (без конечной даты завершения), что приводит к неограниченному росту хранилища и делает решение экономически и технически неэффективным при масштабировании
 - необходимость пересоздания или регенерации задач при изменении правил периодичности (read/write проблема)
 - сложность синхронизации при редактировании задачи: изменение одной задачи требует обновления всех её сгенерированных экземпляров, что может приводить к рассинхронизации данных

2. Scheduler (cron / background worker)
Подход, при котором фоновый процесс (cron/worker) периодически создаёт задачи на основе правил периодичности.

❌ Недостатки:
 - усложнение архитектуры за счёт появления дополнительного сервиса или фонового процесса. При этом scheduler можно выделить в отдельный компонент, что улучшает изоляцию ответственности, но увеличивает общую сложность системы
 - необходимость синхронизации бизнес-логики между API и scheduler’ом
 - потенциальные проблемы с консистентностью данных при рассинхронизации правил и генерации задач
 - увеличение количества инфраструктурных компонентов и точек отказа
 - фактическое дублирование бизнес-логики правил в двух слоях системы (API и worker), что усложняет поддержку и тестирование


3. Rule-based (мой выбор)
Задача хранится как единая сущность с описанием правила периодичности, а её активность вычисляется при запросе.

✔ Преимущества:
 - отсутствие дублирования данных в базе
 - простая и предсказуемая модель хранения (1 задача = 1 запись)
 - гибкость при изменении правил (не нужно пересоздавать данные)
 - отсутствие необходимости в дополнительной инфраструктуре (scheduler/worker)
 - единая точка правды — бизнес-логика в сервисе

Выбранный подход снижает сложность системы и исключает необходимость фоновой синхронизации состояния задач.

🚀 Возможные улучшения

В рамках текущей реализации были приняты упрощённые архитектурные решения.
В дальнейшем систему можно развивать следующим образом:

📦 Модель данных
Разделение таблицы задач на сущности:
 - task
 - task_status (enum-справочник)
 - task_repetition / periodicity

Это позволит:
 - нормализовать данные
 - упростить расширение новых типов периодичности
 - уменьшить связанность полей задачи

🧪 Тестирование
 - Добавить unit-тесты для service layer
 - Покрыть edge-cases правил периодичности
 - Проверить корректность фильтрации по датам

🧠 Архитектура правил
Вынести правила периодичности в отдельные стратегии (strategy pattern / interface-based design)

Это позволит:
 - расширять систему без модификации существующего кода
 - повысить тестируемость
 - изолировать бизнес-логику
